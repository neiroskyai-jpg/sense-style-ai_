# План: правила капсулы и рационального гардероба из курса → в движок

**Задача.** Фаундер дала два набора материалов курса «Алгоритмы имиджа» (папки `капсулы/`
и `капсулы/рациональный гардероб/` на рабочем столе: 53 слайда + 3 расшифровки лекций).
Требование: «учти эти правила в капсуле» — то есть правила должны реально влиять на то,
что генерирует движок, а не лежать документом.

**Что было.** База знаний покрывала алгоритм капсулы и рациональный гардероб частично:
3 типа капсул без вариантов, доли база/тренд, критерии базовой вещи, CPW. Не было главного —
конкретики, которая отличает капсулу стилиста от «набора вещей»: что база, а что тренд по каждому
предмету; что устарело; как собирать комплекты; чем «вещь-сиротка» ломает капсулу.

## Фаза 2. База знаний (`architecture/reference/wardrobe/`) [x]

  мягкая; casual-сумка держит форму, но со скруглёнными краями); комбинаторика «1 низ — максимум

  **база может быть яркой** (2–3 ярких цвета + нейтрали); **личная база зависит от психотипа**
  (экспрессивной клиентке в нейтралях скучно → она не носит такой гардероб); 4 мифа о базе;
  сезонное заимствование палитр внутри температуры; 7 пробелов гардероба; 5 категорий ревизии;
  уточнения чек-листа (3–5 комплектов ИЗ ТЕКУЩЕГО гардероба, игнор размерной сетки, ревизия 3–6 мес).

## Фаза 5. Фикс боевой выдачи Карты (по скриншотам фаундера) [x]

Правила были вшиты в LLM-генерацию, но **визуальная капсула кабинета собирается не моделью, а
отбором из каталога** (`_visual_capsule`) — и туда правила не доезжали. На скриншоте: 4 верхних
слоя и 3 платья при 2 верхах на 3 низа, два одинаковых приталенных жакета, двое одинаковых
расклешённых брюк, «Подъюбник из вискозы» в слоте Низ.

Причина: отбор брал топ-N по релевантности и просто раскладывал по слотам — без квот и дедупа.

- `app/main.py` → `_capsule_quota(n)`: структура капсулы по канону. 6 вещей = 1 верхний слой /
  3 верха / 2 низа; 12 = 2 / 5 / 3 + 1 платье + 1 аксессуар. **Верхов всегда больше, чем низов**
  (добор не может сломать это правило).
- `app/main.py` → `_dedup_products()`: фиды брендов отдают один товар несколько раз; среди дублей
  побеждает packshot (предметное фото).
- `app/main.py` → `_CAPSULE_EXCLUDE` / `_is_capsule_worthy()`: бельё, пижама, подъюбник, топ-бра,
  колготки в ядро капсулы не попадают (по канону расходники в счёт вещей не входят).
- `core/pipeline.py` → `_wearable_hex()` + усиленный `_PALETTE_SYSTEM`: палитра выдавала чистый
  спектр (#0000FF, #FF00FF, #00FFFF, лайм) — таких тканей не бывает. Теперь кислотные оттенки
  приглушаются до текстильных, тон сохраняется; тёмные плотные (изумруд, бордо) и ахроматы
  не трогаются.
- Тесты: `tests/test_capsule_slots.py` +3 (верхов больше низов и ровно N вещей; нет дублей и белья;
  неон гасится, носибельное — нет). Всего 77 passed.

## Фаза 6. Каталог: обувь, сумки, свежие цены [~] — код готов, сбор за фаундером

**Задача:** в каталоге было 48 вещей и НИ ОДНОЙ пары обуви — слот «Обувь» в капсуле пустовал.

**Что выяснили про источники (важно, чтобы не ходить по кругу):**
- **Lamoda парсить нельзя** — сайт отдаёт 403 даже на `robots.txt` (анти-бот). Правильный путь —
  партнёрский XML-фид (Lamoda CPA / Admitad): легально, свежо и даёт партнёрские ссылки
  (комиссия с покупок). Под фид уже готов `core/catalog.py::parse_feed` — нужен только доступ
  фаундера в CPA-сеть.
- Стокманн — 403, Ekonika — 401, Ostin и Mascotte — robots запрещает каталог. Крупные ритейлеры
  закрыты; обход защиты не делаем.
- **Wildberries — рабочий путь:** публичный search-API (v4; v5 отдаёт 429), есть packshot-фото
  и партнёрка.

**Сделано:**
- `scripts/scrape_wb.py` переписан: API v4 + ретраи с backoff; поиск по КАТЕГОРИЯМ под слоты капсулы
  (обувь, сумки, верхняя одежда, верх, низ) + по брендам quiet-luxury; фильтры качества
  (рейтинг ≥ 4.4, отзывов ≥ 10, цена ≥ 1500 ₽, есть в наличии); стоп-лист канона (`base-vs-trend`):
  стразы, потёртости, бахрома, скинни, рюкзаки, бельё, детское; детекция packshot/model по фото
  (переиспользует `scripts/packshot.py`); сводка причин отсева в конце прогона.
- `app/main.py::_catalog_products` — подключён `products_wb.csv` (раньше грузились только Lichi
  и Ushatava, поэтому обувь не появилась бы, даже собранная).
- `core/catalog.py::parse_csv` — читает `style_fields` (иначе вещь не участвует в скоринге по подстилю).
- `app/main.py::_capsule_quota` — в квоту добавлена обувь: 12 вещей = 1 верхний слой / 4 верха /
  3 низа / 1 платье / **2 пары обуви** / 1 сумка. Без каталога обуви капсула деградирует корректно
  (добирает верхами, инвариант «верхов больше низов» держится) — проверено.

**Почему сбор за фаундером:** из среды агента все ритейлеры блокируют по IP (WB — 429, остальные —
403/401). Это ограничение среды, а не кода; в репо это уже было записано (`scrape_brand.py`:
«парсить у себя»). Логика фильтров проверена на синтетических карточках WB.

**Команда для сбора (запускать с домашнего IP):**
```
python scripts/scrape_wb.py                 # категории + бренды → data/fashion-base/products_wb.csv
python scripts/scrape_wb.py --only shoes    # только обувь
```
Если вернёт 0 — смотреть строку «отсеяно»: либо ослабить пороги
(`--min-rating 4.0 --min-feedbacks 0`), либо WB ответил 429 (подождать 5-10 минут).

## Итог

Реализовано целиком. Правила курса теперь влияют на выдачу в трёх точках: knowledge base (RAG) →
системный промпт `look-generator` → hard-rules качества в `pipeline.py`.

Осталось (вне рамок этой задачи):
- `wardrobe-analyzer` (разбор гардероба клиентки: 5 категорий, пробелы, технический прогон) — знание
  под него уже лежит в `rational-wardrobe.md`, самой функции нет.
- Валидация Ксенией: варианты капсул A/B/C и таблица «база vs тренд» — из курса, но стоит сверить
  с её практикой (особенно «тедди устарела», «мини — новая база»).
