# Готовность к клиентам и конкурсу: анонимный доступ, генерация, конструктор, security

Дата: 18.07.2026. Драйвер — конкурс ИТМО (дедлайн 20.07) и первые живые клиентки.

## Задача

Сервис должен работать для клиентки и для жюри **без барьера входа**: прошла квиз — получила
диагностику — получила Карту с образами — попала в кабинет с конструктором из реальных брендов.
Плюс — security-ревью перед публичным запуском.

Решения фаундера (18.07.2026):
- Вход по почте **убираем** как обязательный барьер. Лимит бесплатных генераций перевешиваем
  на устройство (cookie) + IP как страховка.
- Бесплатный тариф = **диагностика по квизу** (текст, дёшево) + **1 генерация с образами**.
  Текстовая Карта по-прежнему не сжигает бесплатную попытку.
- **Квиз не трогаем вообще.** В квизе уже две генерации, и он с самого начала работает без
  почты: анонимный диагноз живёт в `_JOBS` / `data/pending_diag/` и подхватывается на `/card`
  через `from_job` / `last_job` (`app/main.py:3124-3129`). Связка уже есть — менять в квизе
  нечего.

## Контекст кода (на старте)

- Ключ профиля — `TEXT PRIMARY KEY`, не обязательно email (`core/profiles.py:23`). Анонимный id
  ложится в существующее хранилище без миграции схемы.
- Барьер входа стоит **только на `/card` (`app/main.py:3119`) и `/cabinet` (`app/main.py:3262`)**.
  Квиз, диагностика и `/analyze` анонимны уже сейчас. Значит правка точечная, а не сквозная.
- Лимит держится только на email: `count_generations(email) < FREE_GEN_LIMIT` (`app/main.py:3314`),
  а `count_generations` при пустом client возвращает 0 (`core/tracking.py:167`) — то есть
  «просто убрать почту» = безлимитные платные генерации для анонима. Отсюда фаза 1.
- Каталог живой: 486 товаров (`data/fashion-base/`: WB 330, Lichi 131, Ushatava 25),
  `_visual_capsule` (`app/main.py:1455`) собирает капсулу по слотам под палитру/табу/фигуру.
- Админ-доступ работает в обход почты: `?key=SENSE_METRICS_KEY` (`app/main.py:3270`).

## Фазы

### [x] Фаза 1. Анонимная личность и лимиты без почты

- `_current_user()` — единая точка: `session["email"]` если есть, иначе `anon-<uuid4>` из
  постоянной cookie (`httponly`, `samesite=Lax`, `secure` на проде, ~1 год).
- Заменить прямые `session.get("email")` в пользовательских маршрутах на `_current_user()`
  (~20 мест, см. grep). Админские проверки и `/login` не трогаем.
- Лимит генераций: по anon-id (как сейчас по email) + второй контур по IP на сутки
  (`SENSE_IP_GEN_LIMIT`, дефолт подобрать) — на случай чистки cookie.
- Дневная общая квота `DEMO_DAILY_LIMIT` остаётся верхней страховкой.

Сделано: `_current_user()` / `_is_anon()` / `_display_name()` (`app/main.py`), флаги cookie
(HttpOnly, SameSite=Lax, Secure на проде), `record_generation` + `count_generations_ip` и таблица
`gen_log` (`core/tracking.py`), `_client_ip()` + `_ip_gen_allowed()`, лимит `SENSE_IP_GEN_LIMIT`
(дефолт 5/сутки). Тесты — `tests/test_anon_access.py` (9 шт.).

### [x] Фаза 2. Путь без логина

- `/card` и `/cabinet` больше не редиректят анонима на `/login`.
- Почта остаётся **опциональной**: предложение «сохранить результат» после Карты, не барьер.
- Проверить, что нет петли на квиз и что путь только вперёд (канон клиентского пути).

Сделано: `/card`, `/card/build`, `/cabinet`, `/me`, `/stylebook`, `/card/feedback`, `/stylist/msg`
работают без входа. Квиз не тронут. Переходы подтверждены: квиз → `/card?from_job=…`
(`web/identity-scan-quiz.html:2254`), Карта → блок «Следующий шаг» → `/cabinet` (`app/main.py:871`).

### [~] Фаза 3. Проверка вживую

Проверено без обращения к платному API:
- анонимный путь и маршрутизация (`/card` без диагностики → квиз; `/cabinet` без Карты → `/card`);
- оба контура лимита (устройство и IP), текстовая Карта попытку не сжигает;
- конструктор: 483 товара каталога, все с фото и ссылками; капсула собирается на 6 слотов / 12 вещей.

Осталось: прогон с РЕАЛЬНОЙ генерацией на фото — проверить лицо и качество образов (платно).

### [x] Фаза 3a. Качество лица в генерации

Причина «старения» была в самой инструкции: просили скопировать возраст точно и одновременно
требовали `visible texture, pores and fine lines` на плёнке Portra с documentary-подачей — модель
дорисовывала морщины поверх реального лица. Теперь текстура воспроизводится ПО РЕФЕРЕНСУ, добавлен
явный симметричный запрет («ageing her is as wrong as beautifying her»). Тесты —
`tests/test_face_age_prompt.py`. **Требует проверки на живой генерации.**

### [x] Фаза 4. Security-ревью

Прогнать по коду перед публичным запуском. Отдельное внимание:
- секреты и ключи (OPENROUTER_API_KEY, SENSE_METRICS_KEY в query string → логи/реферер);
- админские маршруты `/metrics*` и выгрузки CSV с персональными данными;
- защита от слива токенов при анонимном доступе (фаза 1);
- загрузка фото (тип, размер), XSS в шаблонах, cookie-флаги, 152-ФЗ (журнал согласий).

### [ ] Фаза 5. Тесты и деплой

- Тесты на анонимный путь и на оба контура лимита.
- Полный прогон pytest, затем `git push amvera main:master`, проверка живого домена.

Проведено 18.07.2026. Уязвимостей уровня HIGH/MEDIUM не найдено: анонимные id берутся из uuid4 в
подписанной сессии (подставить чужой нельзя), гейт платного стайлбука сохранён, отзывы на лендинге
экранируются, новый SQL параметризован, SameSite=Lax улучшил защиту от CSRF.

Открытые хвосты (не уязвимости, но чинить надо):
1. **Привязка почты теряет анонимный профиль.** `/auth` (`app/main.py:1337`) переключает личность
   на email, а Карта остаётся под `anon-<hex>`. Клиентка теряет результат ровно тогда, когда
   выполняет нашу просьбу «сохрани». Решение фаундера 18.07: пока оставляем как есть.
2. `X-Forwarded-For` подставляется клиентом, а им же пишется журнал согласий по 152-ФЗ
   (`app/main.py:3222`, раньше был `request.remote_addr`).
3. `anon-<hex>` протекает в шаблон стайлбука (`app/main.py:2569`) — в Карте и профиле закрыто.
