# Кабинет: капсулы (история), сезоны, лист покупок — к ИТМО

**Дата:** 2026-07-02
**Статус:** частично реализован — Кабинет запущен в открытой бете (`/cabinet`), витрина ведёт в него; осталось устойчивость (Фаза 7), packshot/расширение каталога (9–10) и монетизация.
**Связан с:** [[plans/2026-07-02-gap-lifecycle-tiers.md]] (Фаза 3 — трекер разрыва в `/me`)

## Зачем
Фаундер решила довести кабинет (был «СКОРО», теперь — открытая бета) до рабочего к ИТМО (дедлайн 20 июля) и показать его на
сцене. Кабинет = содержание метода: капсульный гардероб (системность) + разрыв во времени (измеримость).

## Решения фаундера (2026-07-02)
- **Формат демо: живая генерация на сцене.** → Значит генерация ОБЯЗАНА быть устойчивой: пополненный
  ключ + грациозный фолбэк (при 402/таймауте показываем последнюю готовую Карту, не сырую ошибку) +
  параллелизм/скорость. Это требование, а не опция.
- **Срез: полный** — капсула с историей (было → обновление) + переключатель сезонов
  (весна-лето / осень-зима) + живой лист покупок.

## Что уже есть (переиспользуем)
- Капсула: `build_style_card` (base_capsule, capsule_board, looks, styling).
- Лист покупок: `generate_shopping_list` (вещи + бюджет; сейчас тизер в Карте).
- Сезон: генерация умеет весна/лето/осень/зима (`_SEASON_LOOK`, param `season`), но Карта захардкожена
  на `FW 2026-2027`.
- Кабинет `/me` + трекер разрыва (Фаза 3).

### Фаза 5 — Сезоны капсульного гардероба [x] (плюмбинг; контент — после ключа)
- `_CARD_SEASONS` (ss=весна-лето / fw=осень-зима), `build_style_card(diag, season=...)` — убран хардкод

- Проброс сезона: анкета Карты (радио-кнопки «Весна–лето / Осень–зима») → `card_build` → воркер;
  текстовый путь `?season=`. Сезон виден в шапке Карты.
- Версии per-season готовы (Фаза 4). Реальный сезонный контент капсулы проверяется после пополнения ключа.

### Фаза 6 — Кабинет: капсула + сезоны + конструктор + лист покупок [x] (база)
- Маршрут `/cabinet` + `CABINET_PAGE`: дашборд по макету фаундера.
- **Конструктор образа** — клиентский: вещи капсулы по слотам, клик/перетаскивание в «Твой образ»
  (по одной вещи на слот), подсветка выбранного, счётчик, «Очистить». Палитра показана как единая
  система цветов («сочетается между собой» = методология выверенной палитры). Проверено кликами в
  Playwright — образ собирается.
- **4 сезона** (весна/лето/осень/зима): вкладки-переключатели, сервер отдаёт капсулу нужного сезона
  из `current_card_by_season`.
- **Лист покупок** — из карты (name + closes_gap).
- Вход из профиля `/me` → «Мой гардероб».
- 59/59 тестов зелёные, дашборд отрендерен и проверен визуально.
- **Осталось (после ключа / отдельно):** фотореалистичная примерка образа «на себе» по кнопке (1
  генерация на выбранную комбинацию); отметки «куплено/ношу» в листе покупок → петля пере-замера;
  карточки истории капсул «было → обновление» (данные Фазы 4 уже есть, нужен UI-блок).

### Фаза 7 — Устойчивость живой генерации (для сцены) [~]

      Тесты `tests/test_gen_fallback.py` (+6): утечка сырого текста, stale vs error, удаление фото.
      Попутно: в `app/main.py` не был импортирован `sys` — `sys.stderr` в except уронил бы NameError
      и задача зависла бы в `processing` навсегда (путь отказа генерации тестами не покрыт вообще).

### Фаза 8 — Точность подбора: палитра↔цвет + стоп [x]
- `core/catalog.py`: `_COLOR_FAMILY_KW` + `_color_families()` — имена палитры метода («Чёрная ночь»,
  «Холодный тауп», «Рубиновый») и сырые цвета фида («чёрный», «тауп», «винный») сводятся к одной
  базовой семье. `match_products`: +2 за цвет в палитре, −1.5 за цвет вне палитры, отсечение по
  стоп-семье (горчичный→yellow и т.п.). Раньше подстрока не находила «Чёрная ночь»↔«чёрный».
- Тесты `tests/test_catalog.py` +2. На реальных данных топ-вещи теперь из палитры. 61/61 зелёный.
- **Осталось:** теги стиля (classic/natural/drama/romance) на вещах фида → подбор под подстиль, не
  только цвет; расширить `_FORMULA_CATEGORIES` на 4 стиля.

### Фаза 9 — Packshot-фото (вещь без модели) [x]
- Запрос фаундера: в конструкторе фото ТОЛЬКО вещи (пиджак = пиджак), без модели и лишних вещей.
- Оказалось, предметные кадры У БРЕНДОВ ЕСТЬ — мы просто брали не тот: Lichi отдаёт модель кадром
  [0], вещь — кадрами [1]/[2]; Ushatava в листинге показывает модельный, а галерея товара лежит на
  странице (`/upload/iblock/...`). Менять бренды не понадобилось.
- `scripts/packshot.py`: кадр предметный, если МАЛО КОЖИ **и** ПЛОСКИЙ ФОН. Одной «кожи» мало —
  Ushatava снимает тёмные вещи на модели без головы: ~0% кожи, притворяется packshot. Ловим по
  разбросу яркости фона (предметный ≈0, студийный с моделью 50–100).
- `scrape_brand.py`: галерея Bitrix группируется по iblock-каталогу, берётся самый мелкий ресайз
  (оригиналы 1920×2547 качались минутами только ради оценки; заодно витрина и PDF легче).
- Каталог: колонка `image_kind` (packshot|model), 33/48 предметных. `_visual_capsule` берёт
  кандидатов с запасом и ставит предметные вперёд → **10/10 вещей капсулы без моделей**; при
  нехватке добирает модельными, а не оставляет слот пустым.
- **Осталось:** 15/48 вещей без предметного кадра в галерее вообще — закроется Фазой 10 (бренды с

### Фаза 9б — Слоты гардероба (баг «жакет и косынка = база») [x]

### Фаза 11 — Капсулы по сценариям жизни (идея фаундера 2026-07-16) [ ]
**Рамка (важно не потерять):** капсулы делятся по **сценариям жизни клиентки**, а НЕ по рубрикам
магазина. Идея пришла от меню 12 STOREEZ («Спортивная капсула», «Пляжная коллекция», «Лен на каждый
день»), но там это мерчандайзинг: нарезка ассортимента, чтобы продать больше. У нас выбор делает
метод по Gap и Формуле, а не клиентка из списка — иначе продукт сползает в «магазин с квизом».
- Сценарий = факт жизни клиентки из анкеты/диагностики (у Анны спорт — это детские тренировки
  полдня, а не рубрика витрины). Основа уже есть: `architecture/reference/scenarios/`
  (`trendy-mama.md`, `home-wardrobe.md`) + [[plans/2026-06-09-glossary-and-scenarios]].
- Показываем только те сценарии, что есть в её жизни: пустая вкладка «вечерняя» у клиентки без
  выходов обесценивает остальное.
- «Базовая» — не сценарий, а ядро капсулы (локомотивы + комплексные из «5 групп товаров»); уже есть.
- Механика — как у переключателя сезонов (Фаза 6), поэтому технически дёшево.
- **Блокер:** каталог. Спортивная/вечерняя капсула из нынешней базы (Lichi romance/drama + Ushatava,
  без обуви и сумок) = пустой экран. Сначала WB-канал и Фаза 10, потом сценарии.
- **Сроки:** после 20 июля. За 4 дня до подачи новая фича ломает работающее демо ради того, чего
  жюри не просило.
