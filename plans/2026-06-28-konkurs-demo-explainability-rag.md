# План: Конкурсное демо — объяснимость + 2 направления + настоящий RAG

**Цель:** довести демо до уровня, который читается жюри ИТМО/AI Hub как «AI-система для диагностики визуальной идентичности», а не «сайт стилиста». Три недостающих под жюри элемента: объяснимость (Explainable AI), два именованных направления образа, настоящий retrieval (pgvector).
**Связан с:** [2026-06-25-mvp-vertical-slice.md](2026-06-25-mvp-vertical-slice.md), [2026-06-25-itmo-jmlc-zayavka.md](2026-06-25-itmo-jmlc-zayavka.md)

## Контекст: что УЖЕ есть (не переделываем)

Сквозной пайплайн фото→vision→диагностика→капсула→рендер образов на клиентке, Identity Gap %, трекинг динамики, eval-судья, async job_id, деплой на Amvera (РФ), согласия 152-ФЗ. Квиз `web/identity-scan-quiz.html` считает «направление» по 12 словам 4 семантических полей и зовёт `/api/analyze`.

Дельта под конкурс — **подача и объяснимость на экране результата + retrieval-движок под ней**, не стройка с нуля.

## Решения (2026-06-28)

- **Приоритет:** сначала видимый экран результата + объяснимость → потом RAG (грунтует «почему») → потом экран архитектуры (отражает реальный RAG). Решение делегировано Claude, обосновано: видимость × риск × синергия.
- **RAG:** строим **настоящий** (pgvector в Supabase), не «честную подачу промпт-слоя». Выбор фаундера. Корпус — `architecture/reference/` (цветотипы, фигуры, подстили, сценарии, лексикон впечатления).

## Фаза 1. Экран результата + объяснимость [ ]

Цель: после квиза пользователь видит «что AI увидел по фото», «почему AI так решил» (3 сигнала), 2 именованных направления с «подходит, если…», «попробуй сейчас», стоп-лист.

- [x] **Бэкенд:** `_run_fast` ([app/main.py](../app/main.py)) теперь отдаёт `directions[]` + `explain{}`; `_explainability()` строит сигналы/фото/стоп-лист из `diag` без лишних LLM-вызовов.
- [x] **Два направления:** `generate_directions()` в [core/pipeline.py](../core/pipeline.py) — один дешёвый текстовый вызов (grounded в диагностике, tone of voice): 2 направления {name, fits_if, items[], image_generation_prompt}. Рендер обоих параллельно на клиентке.
- [x] **«Почему AI так решил»:** блок из 3 сигналов (want_traits + топ семантические поля + now→разрыв) + `dna_explanation`.
- [x] **«Что AI увидел по фото»:** чипы контраст/цветотип/фигура/силуэт + риск считывания (из `diag`).
- [x] **Стоп-лист «что не покупать»:** из `visual_formula.stop_list` (топ-4).
- [x] **Фронт квиза:** `renderAIResult()` + секция `#r-ai-extra` в [web/identity-scan-quiz.html](../web/identity-scan-quiz.html); `pollResult` сохраняет полный `aiResult`. Дубль образов в pain-card убран при наличии направлений.
- [ ] **Форма /demo:** отложена (вторичный путь, идёт через тяжёлую капсулу; уже показывает формулу/gap/dna).
- [ ] **Сквозной тест на реальном фото** (нужен платный прогон — по согласию фаундера): видны 2 направления + объяснимость + образы. Проверено без API: тесты зелёные (13), Flask поднимается, квиз отдаёт новую секцию.

## Фаза 2. Настоящий RAG (локальный, pgvector-ready) [x]

Цель: диагностика и объяснимость опираются на retrieval из векторной базы правил. Жюри видит retrieval + ссылки на сработавшие правила.

**Решение (2026-06-28):** локальный RAG вместо Supabase pgvector. Корпус крошечный (~256К символов), локальные эмбеддинги = «настоящий RAG» без инфраструктуры, бесплатно, РФ-комплаентно (ничего не уходит вовне). Интерфейс `retrieve()` готов к переключению на pgvector, когда база вырастет.

- [x] **Корпус → чанки + теги:** [scripts/build_rag_index.py](../scripts/build_rag_index.py) нарезает `architecture/reference/` (12 папок) на 130 чанков; авто-теги цветотип/фигура/поле по enum'ам диагностики (теги по заголовку секции — без ложных кросс-матчей). 64 чанка с тегами, покрыты все 12 цветотипов / 5 фигур / 4 поля. Закоммичено: `data/rag/chunks.json`.
- [x] **Эмбеддинги + хранилище:** локально fastembed (ONNX, без torch), модель `paraphrase-multilingual-MiniLM-L12-v2` (384 dim, RU). Вектора `data/rag/vectors.npz` (185 КБ) закоммичены. fastembed/numpy — опциональны (в `requirements.txt` закомментированы); без них retrieval работает по тегам.
- [x] **Retriever:** [core/rag.py](../core/rag.py) — гибрид: точная фильтрация по тегам (всегда) + cosine по эмбеддингам (когда есть). Приоритет канона над трендами, разнообразие по источникам, мягкий фолбэк.
- [x] **Встроено в диагностику:** [core/pipeline.py](../core/pipeline.py) `diagnose()` грунтует промпт правилами под ранние сигналы (цветотип/фигура/запрос) + прикрепляет `retrieved_rules` по итоговому профилю. RAG-сбой не роняет диагностику.
- [x] **Грунтовка объяснимости:** блок «Из авторской базы знаний сработали правила» в квизе (`renderAIResult`) — теги-источники + цитаты ([web/identity-scan-quiz.html](../web/identity-scan-quiz.html)).
- [x] **Тесты:** [tests/test_rag.py](../tests/test_rag.py) — 5 тестов (релевантность, отсутствие кросс-матчей фигур, форма cited, фолбэк). Весь набор: 18 зелёных.
- [ ] **Валидация тегов Ксенией** (~15 мин, её экспертиза) — провалидировать авто-теги чанков.
- [ ] **Живой прогон** (платный, по согласию) — увидеть RAG-правила в реальном результате.
