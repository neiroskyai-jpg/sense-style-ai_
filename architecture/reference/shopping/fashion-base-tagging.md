# Методология разметки умной fashion-базы (бренды × вещи)

**Что это.** Единый стандарт разметки «умной» базы: каждый **бренд** и каждая **вещь** описываются не ценой и ссылкой, а стилем, силуэтом, цветотипом, ситуацией жизни, капсульностью и эффектом на фигуру. Это язык, на котором движок подбирает под профиль клиентки: **бренд → вещь → стиль → фигура → цветотип → ситуация → капсула → рекомендация**.

**Статус:** v0.1 (2026-07-01). Схема + правила разметки. Реализация по плану `plans/2026-07-01-smart-fashion-base-tagging.md`.

**Зачем отдельный документ.** Есть два соседа, которые это методология связывает:
- `brand-matrix.md` — бренды × бюджет × стилевое поле (уровень бренда). Этот документ его расширяет и задаёт словари.
- `core/catalog.py::Product` — техническая нормализация вещи из фида (уровень SKU). Этот документ говорит, **какими смысловыми полями** дополнять `Product` и **откуда** их брать.

---

## 0. Главный принцип — метод не переписываем, а надстраиваем

Разметка **обязана** говорить на языке канона метода, иначе подбор развалится:

- Стиль — **4 базовых поля** (`classic / drama / romance / natural`) + **25 подстилей** из [`sense-style-method.md`](../../sense-style-method.md) §4–5. Никаких параллельных таксономий.
- Фигура — **коды движка** `rectangle / pear / apple / hourglass / inverted_triangle` (внутри), геометрия на выходе — [`modern-figure-language-2026.md`](../figure-correction/modern-figure-language-2026.md). Никаких «груша/яблоко» в тексте клиентке.
- Цвет — **12-сезонная система** цветотипов из [`colortypes/`](../colortypes/12-cvetotipov.md). Табу-цвет цветотипа — жёсткий фильтр, его тренд не перебивает.
- Силуэт — словарь из конспекта «Силуэты»: `прямой / полуприлегающий / прилегающий(X) / трапеция / овал`.

> **Тренд ≠ эвергрин.** Стиль/фигура/цветотип — эвергрин-фильтр. Тренд — акцент **внутри** фильтра. Разметка тренда без привязки «кому по фигуре · в какой палитре · до какого сезона» в базу не идёт (см. [`knowledge-and-vision-architecture.md`](../../2026-06-09-knowledge-and-vision-architecture.md) §1–3).

## 1. Два уровня разметки

| Уровень | Единица | Темп изменений | Кто заполняет |
|---|---|---|---|
| **Бренд** | марка целиком | медленно (сезон-полугодие) | куратор (Ксения) + веб-поиск |
| **Вещь (SKU)** | конкретный товар | быстро (наличие/цена) | фид (авто) + Claude Vision (авто) + добор куратором |

Правило разделения (из `brand-matrix`): **бренд** привязан к бюджету и стилевому полю; **палитра и силуэт фильтруются на уровне вещи** — любой бренд делает вещи всех цветов. Не размечай цвет на бренде.

## 2. Схема уровня `brands`

Расширяет строку из `brand-matrix.md`. Минимум для MVP помечен ⭐.

| Поле | Тип / словарь | Пример | Источник |
|---|---|---|---|
| `brand_name` ⭐ | строка | `12 STOREEZ` | куратор |
| `segment` ⭐ | `low / middle / high / luxury` | `high` | куратор |
| `style_fields` ⭐ | 1–2 из `classic/drama/romance/natural` | `[classic, natural]` | куратор |
| `substyles` | коды §5 (см. §4) | `[minimalism, quiet_luxury]` | куратор |
| `strong_categories` ⭐ | список категорий | `[трикотаж, пальто, брюки]` | куратор |
| `best_for_use_cases` | словарь §7 | `[office, city, travel]` | куратор |
| `visual_code` | короткая фраза (ToV) | `спокойный, лаконичный, статусный` | куратор |
| `site` / `marketplace` / `social` ⭐ | ссылки | — | куратор |
| `source` | `курс / веб / вручную` | `веб` | куратор |
| `status` ⭐ | `проверить / активен / неактивен` | `проверить` | куратор |
| `verified_at` ⭐ | дата | `2026-07-01` | куратор |

**Нормализация seed-списка** (бренды из «ГИД ПО НАСМОТРЕННОСТИ АИ» и «Алгоритм создания капсулы»): убрать дубли → отделить бренды от шоурумов/магазинов → проверить, что жив → присвоить `segment` + `style_fields` → поставить `status: проверить`. Готовое ядро уже есть в `brand-matrix.md` — его валидируем, а не собираем заново.

## 4. Словарь стилей — как свести коды в канон

Пользовательские «исторические/коммерческие коды» (preppy, boho, minimalism, athleisure, lady like…) **не отдельная система** — они ложатся на 4 поля и 25 подстилей. Таблица маппинга (используй `substyle`-код, если он есть; иначе — свободный `style_codes`-тег для насмотренности):

| Расхожий код | Поле | Подстиль метода (§5) |
|---|---|---|
| minimalism | classic | `minimalism` (№4) |
| preppy | classic | `preppy` (№6) |
| parisian chic | classic | `parisian_chic` (№7) |
| dandy / garcon | classic | `dandy_garcon` (№3) |
| lady like | romance | `lady_like` (№17) |
| slip / бельевой | romance | `slip_dress` (№16) |
| boho / boho-chic | natural | `boho` (№20) |
| athleisure / sport chic | natural | `sport_chic` (№19) |
| casual chic / smart casual | natural×classic | `smart_casual` (№25) |
| quiet luxury | classic×natural | `quiet_luxury` (№23) |
| power woman / power tailoring | classic×drama | `power_woman` (№24) |
| rustic / country / cottagecore | natural | `rustic` (№22), `english_country` (№21) |
| grunge / деконструкция | drama | `deconstruction` (№12) |
| military, safari, neo-futurism | — | **нет канон-подстиля** → свободный `style_codes`-тег, к полю привязать по маркерам (обычно natural/drama) |

**Правило «современного прочтения»** (из «Коммерческие и исторические стили»): стиль в базе — это набор признаков (крой, фактура, цвет, принт, декор, длины), а не ярлык «в лоб». Для микса — 1–2 поля, максимум 3 точечно; отмечай риск перестилизации у бренда полем `visual_code`/заметкой, а не отдельным жёстким полем в MVP.

## 5. Правила «силуэт → фигура» (как заполнять `best_for_figures`)

`best_for_figures` не размечается вручную на глаз — **выводится** из силуэта/кроя по методу коррекции фигуры ([`figure-correction.md`](../figure-correction/figure-correction.md)). Стартовый ruleset (расширяется):

| Задача фигуры | Что в вещи | Кому (коды движка) |
|---|---|---|
| нужна талия | X-силуэт, пояс, посадка по талии | `rectangle`, `apple` |
| смягчить линию плеч | мягкое плечо, без объёма сверху | `inverted_triangle` |
| уравновесить бёдра | объём/акцент сверху, прямой низ | `pear` |
| вытянуть силуэт | вертикали, монохром, высокая посадка | все |
| сохранить баланс | полуприлегающий, естественная талия | `hourglass` |

Антипаттерны (жёсткое исключение при подборе, уже заложено в `catalog.py::_FIGURE_AVOID`): напр. `rectangle` ← футляр/балахон/оверсайз-без-пояса. Каждый новый антипаттерн — со ссылкой на правило метода, не «на вкус».

## 6. Капсульность и рациональность (как заполнять `capsule_role`, `mixability_score`)

Из «Алгоритм создания капсулы» и «Рациональный гардероб»:

- Капсула = 6–12 вещей; каждая новая вещь должна миксоваться минимум с **3–4** уже имеющимися. Отсюда `mixability_score` 0–3 (сколько категорий-компаньонов у вещи): база/нейтральный цвет/простой крой → 3; акцентный принт/сложный цвет → 1.
- Структурный перекос в пользу верха: **верхов больше, чем низов**. Целевая пропорция капсулы: `2 низа + 4–5 верхов + 1–2 верхних слоя + 2 пары обуви + аксессуары`. Движок при сборке капсулы соблюдает её — не выдаёт 5 брюк и 2 верха.
- `capsule_role` = роль в этой структуре (база/акцент/верхний слой/обувь/аксессуар). Из неё же считается вклад в `mixability_score`.
- Рациональные скоры (опционально, фаза 2): `variability_score`, `quality_score`, `outfit_count_potential`, `status_fit` — критерии оценки пользы вещи из «Рационального гардероба».

Шаблоны капсул (`capsule_templates`) — отдельная короткая таблица под `use_case` (casual / office / home / mama_walk / evening), см. §7.

## 7. Ситуации жизни — словарь `use_case`

Из «Trendy mama» и «Домашний гардероб». Это то, чего нет у большинства fashion-сервисов — сильная зона.

| `use_case` | Требования к вещи |
|---|---|
| `office` | статус, структура, smart casual |
| `city` | комбинаторика, комфорт, немаркость |
| `mama_walk` | тепло, немаркость, многослойность, лёгкость снять/надеть, минимум мешающих деталей, свободный/полуприлегающий силуэт |
| `mama_cafe` | smart casual, чуть наряднее |
| `home` | мягкость, дышащие ткани, не сковывает движение, по сезону (сон / спорт / встреча гостей — подкатегории) |
| `travel` | немнущиеся ткани, удобная обувь, высокая капсульность |
| `evening` | акцентность, нарядность, вечерний код |

## 8. Диагностика запроса (перевод «хочу красиво» в фильтр базы)

Из «Типы запросов у клиентов». Живёт не в базе вещей, а в слое диагностики — но определяет, **что искать**. AI переводит неконструктивный запрос в конструктивный → в поля фильтра:

| Исходный запрос | Конструктивно | Фильтр базы |
|---|---|---|
| «шкаф полный, надеть нечего» | нужна рациональная капсула под образ жизни | `capsule`, высокий `mixability_score` |
| «хочу выглядеть дорого» | статусный smart casual / quiet luxury | `substyle: quiet_luxury/smart_casual`, `segment: high` |
| «я в декрете, но хочу быть стильной» | удобная красивая mama-капсула | `use_case: mama_walk`, поле `natural/classic` |
| «хочу больше женственности» | мягкие силуэты, платья, юбки | поле `romance`, `silhouette: X` |

Стыкуется с существующим `prompts/request-diagnostics` и базами выбора стилей (`style-selection-cards.md`).

## 9. Конвейер разметки (кто что заполняет)

1. **Фид (авто).** `catalog.py::parse_feed` → технические поля `Product` (name/brand/price/color/sizes/url/image/in_stock). Дёшево, обновляется по расписанию.
2. **Наследование от бренда (авто).** `style_fields`, `segment`, `best_for_use_cases` вещь берёт от своего бренда как дефолт.
3. **Claude Vision (авто по фото товара).** Силуэт, fit, line_type, плечо, объём, температура/глубина/яркость цвета, `capsule_role`. Один промпт «фото вещи → поля разметки» (по образцу `prompts/vision-analyzer.md`, но для товара, а не клиентки).
4. **Вывод по правилам (авто).** `best_color_types`/`avoid_color_types` — из color-полей; `best_for_figures`/`emphasizes/hides` — из силуэта (§5); `mixability_score` — из роли/цвета (§6).
5. **Добор куратором.** `is_trend` + сезон-срок, спорные стилевые коды, валидация Vision-разметки на первых партиях.

Приоритет для MVP: сначала уровень бренда (§2) на валидированном ядре из `brand-matrix`, затем разметить **20–30 вещей** под первый срез (напр. `rectangle × natural`), проверить подбор `catalog.py::match_products`, потом масштабировать через Vision.
